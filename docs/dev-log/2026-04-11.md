## 의사결정

### 노트 삭제 시 대상의 id

- 고민:
  - `handleDeleteNote`에서 삭제한 노트의 id로 어떤 걸 사용할지

- 선택지:
  1. `deleteNote`의 응답으로 받은 id
  2. `selectedNoteId`

- 판단 기준:
  - Source of Truth

- 결정:
  - `selectedNoteId` 사용

- 이유:
  - delete는 서버가 값을 "변형하지 않음"  
    즉 서버가 뭔가 새로 만들어주지 않고, 그냥 없애기만 함
  - 그래서 클라이언트가 이미 충분한 정보(삭제할 대상의 id)를 알고 있으므로 입력값 그대로 사용 가능함

- create/update는 서버 결과를 쓰는 이유:
  - 서버가 id 생성 등 데이터의 최종 상태를 결정하기 때문에
  - 클라이언트 정보는 불완전해 반드시 서버 결과를 사용해야 함

- Source of Truth:

  데이터의 최종 상태를 결정하는 주체

  |     작업      | source of truth |
  | :-----------: | :-------------: |
  |    delete     |     client      |
  | create/update |     server      |

- 트레이드오프:
  - delete에서도 서버 응답 사용으로 통일하면 일관성은 높아질 수 있음
  - 하지만 현재 mock API에서는 서버가 추가 정보를 반환하지 않기 때문에 클라이언트 값 사용이 단순하고 명확함

- 예외:
  - 서버에서 삭제 실패 시 (예: 이미 삭제된 데이터)
  - 클라이언트 상태와 서버 상태가 불일치할 가능성

- 결론:
  > 서버가 값을 **결정**하는 작업인지, **처리만** 하는 작업인지 구분한다.
