# 2026-04-15

## 오늘 작업

- 노트 API 비동기 작업별로 loading 상태 세분화
- 작업별로 독립적인 UI 표현
  - 전체 로딩 → overlay
  - 저장 → 저장 버튼 disable
  - 삭제 → 삭제 버튼 disable

### 문제 상황

- 초기 로딩(getNotes) 중 동시에 저장하면, 둘 다 `loading = true`가 되어 UI로 구분을 못 함
- 즉 하나의 loading은 여러 비동기 상태를 표현하기에 부족함

### 해결 전략

각 비동기 작업을 독립적으로 표현

1. 상태를 객체로 변경
2. 작업별 상태 업데이트 패턴 변경
3. 독립적 UI 반영

### 작업의 핵심 포인트

```
loading을 “비동기 작업 단위”로 분리
→ 동시에 여러 loading 가능
→ 독립적 UI 반영 가능 → UX 향상
```

## 의사결정

### loading 세분화 방식

- 고민:
  - 비동기 작업(fetch, save, delete)이 동시에 발생할 수 있는 상황에서 각각의 상태를 어떻게 표현할 것인지
  - 하나의 loading으로는 서로 다른 작업 상태를 구분할 수 없음

- 선택지:
  1. 하나의 loading 객체 안에서 분리  
     (`{ fetch, save, delete }`)
  2. 여러 개의 useState로 분리  
     (`fetchLoading`, `saveLoading`, `deleteLoading`)

- 판단 기준:
  - 상태 관리 일관성
  - UI와의 연결 명확성
  - 상태 확장성

- 결정:
  - 하나의 loading 객체 안에서 작업별로 분리

- 이유:
  - 관련된 상태를 하나의 객체로 관리하면, 상태 구조를 한눈에 파악할 수 있고 일관성 유지가 쉬움
  - UI에서도 loading.save, loading.delete처럼 명확하게 매핑할 수 있음
  - 추후 loading 종류가 늘어나도 구조 확장이 용이함
  - 함수형 업데이트를 통해 일부 상태만 안전하게 변경 가능

- 트레이드오프:
  - 상태 업데이트 시 spread 문법을 사용해야 해서 코드가 다소 길어짐
  - 객체 구조를 잘못 관리하면 특정 상태가 덮어씌워질 위험이 있음

- 맥락:
  - 현재는 작업 수가 적지만, 이후 비동기 작업이 늘어날 가능성을 고려해 확장 가능한 구조를 우선 선택

- 영향:
  - 모든 비동기 로직에서 함수형 업데이트 패턴 적용
  - UI에서 참조할 loading 상태를 작업 단위로 변경
  - 버튼 disable 및 overlay 처리 → UI 로직이 더 명확해짐

- 결론:
  > loading 상태는 컴포넌트 단위가 아니라 “비동기 작업 단위”로 설계해야 한다.
