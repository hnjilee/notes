## 오늘 작업

- 노트 API 낙관적 업데이트 적용
- loading 상태의 의미를 pending으로 재정의

### 현재 문제

느린 UX

```
사용자 행동 → 대기 → API 성공 → state 반영
```

### 해결 방법

낙관적 업데이트

```
state 먼저 변경 → API 요청 → 실패 시 롤백
```

### 작업의 핵심 포인트

낙관적 업데이트 이후의 loading은 “작업 대기”가 아니라 “확정 대기”를 표현해야 한다.

- 기존 loading 의미: 작업 중이라 아직 UI 반영 전
- 낙관적 업데이트 이후 loading 의미: UI는 반영됐지만 서버 확정 대기 중

## 의사결정

### 낙관적 업데이트와 loading 상태의 관계

- 고민:
  - 낙관적 업데이트를 적용하면 UI가 먼저 반영되는데, 기존처럼 `loading.save`, `loading.delete`를 “작업 중” 상태로 사용하면 이미 반영된 UI와 충돌이 발생함
  - 사용자 입장에서는 작업이 끝난 것처럼 보이는데, 여전히 “저장 중...” / “삭제 중...”으로 표시되는 것이 어색함

- 선택지:
  1. 기존 방식 유지 (loading을 작업 중 상태로 그대로 사용)
  2. 낙관적 업데이트 시 save/delete loading 제거
  3. loading의 의미를 “작업 중”이 아닌 “서버 확정 대기 상태”로 재정의

- 판단 기준:
  - 사용자 경험의 일관성 (UI와 상태 표현의 불일치 여부)
  - 중복 요청 방지 가능 여부
  - 상태 일관성

- 결정:
  - fetch는 기존처럼 “로딩(차단)” 상태로 유지
  - save/delete는 “서버 확정 대기(pending)” 상태로 재정의

- 이유:
  - 낙관적 업데이트에서는 UI가 먼저 반영되므로, save/delete를 단순 로딩 상태로 표현하는 것은 의미가 맞지 않음
  - 하지만 서버 요청이 완료되지 않았기 때문에 중복 요청 방지와 상태 일관성을 위해 상태 자체는 유지해야 함
  - 따라서 loading을 제거하는 것이 아니라 의미를 pending 상태로 재해석하는 것이 적절하다고 판단함

- 트레이드오프:
  - loading이라는 이름과 실제 의미(pending)가 완전히 일치하지 않음
  - (대안) status/pending 등으로 명확히 분리할 수 있지만 현재 단계에서는 구조 복잡도를 증가시키지 않기 위해 유지

- 영향:
  - save/delete 버튼은 “차단”이 아니라 “중복 요청 방지” 용도로 사용
  - 버튼 텍스트 및 UI 의미를 “대기” → “동기화”로 변경

- 결론:
  > 낙관적 업데이트 이후의 loading은 “작업 중”이 아니라 “서버 확정 전 상태(pending)”로 해석해야 한다.
