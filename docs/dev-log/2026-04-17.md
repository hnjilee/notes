# 2026-04-17

## 오늘 작업

- 노트 API 에러 global/form으로 분리 및 UI 반영
- retry 흐름 구현
  - fetch: fetchNotes로 분리 → 재요청
  - save/delete: 실패 시 retryAction에 작업 저장 → 재실행

### 현재 문제

fetch/save/delete 에러 전부 같은 위치

- 어디서 발생했는지 불명확
- 사용자 행동 유도 불가

### 해결 전략

- error는 하나가 아니라 범위별로 나눠야 하고,
- 그 기준은 “사용자가 어디를 보고 있는가”
- retry로 실패한 작업을 다시 실행할 수 있는 방법 제공

```
global error: fetch (페이지 전체에 시선)
form error: save/delete (에디터에 시선)
```

## 의사결정

### 삭제 에러 분류

- 고민:
  - 노트 API 에러를 global/form으로 분리하면서 삭제 에러를 어느 범위에 포함시킬지
  - 삭제는 리스트 데이터에 영향을 주기 때문에 global로 볼 수도 있고, 삭제 버튼은 Editor에 있기 때문에 form으로 볼 수도 있음

- 선택지:
  1. global error
     - 리스트 데이터 변경이므로 전체 상태 문제로 간주
  2. form error
     - 삭제 액션이 Editor에서 발생하므로 해당 컨텍스트에 포함

- 판단 기준:
  - 사용자의 시선 위치
  - 액션이 발생한 컨텍스트
  - 에러 인지 흐름의 자연스러움

- 결정:
  - form error

- 이유:
  - 사용자는 삭제 버튼을 Editor에서 클릭한 후 Editor를 보고 있는 상태에서 결과를 인지하게 됨
  - 낙관적 업데이트로 인해 List에서 노트가 먼저 사라졌다가 실패 시 롤백되는 UI를 확인한 뒤 List 옆에 있는 Editor에서 에러 메시지를 보는 것이 자연스러운 흐름이라고 판단
  - 따라서 데이터 영향 범위보다 사용자의 시선과 행동 흐름을 우선 기준으로 선택

- 트레이드오프:
  - 삭제는 실제로 리스트 데이터에 영향을 주기 때문에 global error로 분류하는 것이 더 직관적일 수 있음
  - 하지만 이 경우 사용자의 시선 이동이 발생하고 에러 인지 흐름이 끊길 수 있음

- 영향:
  - 삭제 에러 메시지를 Editor 영역에 표시
  - 낙관적 업데이트 → 롤백 → 에러 확인까지의 흐름이 하나의 컨텍스트에서 유지됨
  - global error는 fetch와 같은 “페이지 단위 문제”로 역할이 명확히 분리됨

- 결론:
  > 에러는 상황에 따라 데이터 영향 범위가 아니라  
  > “사용자의 시선과 행동 흐름”을 기준으로 분류할 수 있다.
