## 중복 상태 제거

### 현재 문제

`selectedNoteId` / `draftNote.id`

→ 같은 의미 두 개  
→ sync 문제 발생 가능

### 해결 전략

Source of Truth 최소화 → 나머지는 파생

> 이 상태는 진짜 독립 상태인가, 아니면 파생 가능한가?

### 의사결정

- 선택지:
  1. 현재 구조 유지
  2. selectedNoteId 제거

- 결정:
  - draftNote를 단일 source로 사용

- 이유:
  - 실제 UI는 draft 기준: Editor → draftNote 사용
  - 중복 상태 제거로 sync 문제 예방

- 트레이드오프:
  - 선택 상태 vs 편집 상태 간 구분의 명시성 감소
  - 개발 초기에는 흐름 파악을 위해 상태 역할을 명확히 분리함

- 효과:
  - 상태 최소화
  - source 명확화
  - 상태 간 관계 제거 → 의존 줄이기

## CATEGORY 상수 분리

### 현재 문제

categories → notes 기반 동적 생성

- 동적이고 유연하지만,
- 오타 위험, 변경 어려움, 의미 불명확
- 데이터 오염 시 UI도 함께 깨질 수 있음

### 해결 전략

상수 기반으로 전환 → 의미가 명확하고 통제 가능해짐

> 값을 직접 쓰지 말고, 이름/개념으로 다루기

### 의사결정

- 선택지:
  1. 단순 객체
  2. value + label 분리

- 결정:
  - value/label 구조로 관리

- 이유:
  - 데이터(value)와 UI(label) 분리
  - 확장 가능

- 효과:
  - label을 같이 갖고 있어서 UI에 표현 가능해짐
  - 단순 문자열이 아니라 의미 있는 이름으로 관리해서 명확해짐
  - source가 일원화되어 통제 및 관리가 용이해짐

- 트레이드오프:
  - 단순 객체로 정의하면 간단하고 직관적임
  - 중첩 객체 구조는 로직이 복잡해질 수 있음

- 추가 결정:
  - CATEGORY를 특정 hook(useNotes)에 두지 않고 별도의 constants 파일로 분리
  - 이유: 도메인 값은 특정 로직이 아니라 전체 UI에서 공유되는 개념이기 때문
