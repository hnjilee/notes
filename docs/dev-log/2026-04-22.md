## 로딩 최소 표시 시간 구현

### 현재 문제

```
API가 너무 빠르면
→ 로딩 UI가 깜빡임 / retry 클릭 시 무반응처럼 보임
→ UX 불안정
```

### 해결 전략

로딩 최소 300ms 유지

```
요청 시간 측정
→ 부족한 만큼만 delay / 실제 요청이 길면 그대로 유지
→ UX 안정화
```

### 의사결정

- 고민:
  - API 응답이 빠를 경우 로딩 UI가 매우 짧게 표시되어 깜빡이거나, retry 클릭 시 반응이 없는 것처럼 보이는 UX 문제 발생
  - 로딩을 인위적으로 지연할 경우 전체 응답 속도가 느려질 수 있음

- 선택지:
  - 로딩 delay를 고정값으로 추가 (항상 추가적으로 일정 시간 지연)
  - 요청 시간에 따라 부족한 만큼만 delay 추가

- 판단 기준:
  - UX 안정성
  - 실제 응답 속도 저하 여부
  - 사용자 체감 성능

- 결정:
  - 요청 실행 시간을 기준으로 부족한 만큼만 delay를 추가하는 방식 선택

- 이유:
  - 고정 delay는 항상 응답을 느리게 만들어 UX를 저하시킬 수 있음
  - 요청 시간을 기준으로 하면 빠른 요청만 보정하고, 느린 요청에는 영향을 주지 않음
  - 로딩이 최소한의 시간 동안 유지되면서도 실제 성능은 유지 가능

- 트레이드오프:
  - 구현 복잡도 증가 (시간 측정 + 조건부 delay)
  - 로딩 종료 시점이 즉각적이지 않고 일정 시간 유지됨

- 영향:
  - fetch/save/delete 모든 비동기 작업에서 로딩 UI가 안정적으로 유지됨
  - retry 시 무반응처럼 보이는 문제 개선
  - 낙관적 업데이트 이후 “동기화 과정”이 더 자연스럽게 표현됨

- 결론:
  - 로딩은 단순히 빠르게 사라지는 것이 아니라, 사용자에게 안정적으로 인지될 수 있어야 한다.

## 로딩 텍스트 표현 방식

- 고민:
  - 저장/삭제 각각의 행위를 드러내기 위해 “저장 중 / 삭제 중”으로 표시할지,
  - 하나의 표현으로 통일할지 결정 필요

- 선택지:
  - “저장 중 / 삭제 중” (행위 중심 표현)
  - “동기화 중” (상태 중심 표현)

- 판단 기준:
  - 낙관적 업데이트 적용 여부
  - UI와 텍스트 간의 일관성
  - 사용자 인지 흐름

- 결정:
  - “동기화 중”으로 통일

- 이유:
  - 현재 구조는 낙관적 업데이트로 인해 UI가 먼저 변경되고, 이후 서버와 상태를 맞추는 흐름
  - 이 경우 사용자는 이미 저장/삭제가 완료된 것처럼 인지하기 때문에, “저장 중 / 삭제 중”보다 “동기화 중”이 실제 상태를 더 정확하게 설명함

- 트레이드오프:
  - 개별 액션(save/delete)에 대한 명확성은 감소할 수 있음
  - 초기 사용자에게는 표현이 다소 추상적으로 느껴질 수 있음

- 영향:
  - 로딩 텍스트가 전체적으로 일관된 개념으로 통일됨
  - 낙관적 업데이트 UX와 텍스트 간의 불일치 해소
